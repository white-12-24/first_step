# Bible RAG Chatbot

성경 본문을 기반으로 질문에 답변하는 **근거 기반 RAG(Retrieval-Augmented Generation) 챗봇** 프로젝트입니다.  
사용자의 질문을 분석한 뒤 관련 성경 구절을 검색하고, 검색된 본문을 근거로 답변을 생성하는 구조로 설계했습니다.

단순히 LLM에게 자유롭게 답변을 맡기는 방식이 아니라,  
**성경 본문 데이터 → 전처리 → 청크 생성 → 벡터 검색 → 근거 기반 답변 생성** 흐름을 구현하는 데 초점을 두었습니다.

---

## 1. 프로젝트 개요

### 프로젝트 목적

성경 관련 질문에 대해 LLM이 임의로 답변하는 문제를 줄이고,  
실제 성경 본문에서 검색된 근거를 바탕으로 답변하는 챗봇을 구현하는 것이 목표입니다.

특히 다음과 같은 문제를 해결하고자 했습니다.

- LLM이 성경 구절을 잘못 인용하거나 존재하지 않는 내용을 말하는 문제
- 질문과 관련 없는 본문을 근거로 답변하는 문제
- 검색 결과가 부족한 상황에서도 억지로 답변을 생성하는 문제
- 사용자가 질문한 주제와 성경 본문 사이의 연결성을 높이는 문제

---

## 2. 핵심 기능

### 1) 성경 본문 수집 및 정제

Wikisource의 성경 본문 데이터를 기반으로 성경 구절 데이터를 수집했습니다.  
수집된 원문은 장, 절, 본문 단위로 정리하고, 검색에 활용할 수 있도록 전처리했습니다.

주요 처리 내용은 다음과 같습니다.

- 성경 권별 본문 수집
- 장/절 단위 구조화
- 중복 구절 확인
- 빈 본문 제거
- 검색용 텍스트 정규화
- CSV 형태로 저장

---

### 2) 청크 데이터 생성

RAG 검색 성능을 높이기 위해 성경 본문을 일정 단위로 묶어 청크 데이터를 생성했습니다.

단일 구절만 검색할 경우 문맥이 부족할 수 있기 때문에,  
인접 구절을 함께 포함하는 방식으로 질문과 더 잘 연결되는 검색 단위를 구성했습니다.

예시 구조는 다음과 같습니다.

| 컬럼명 | 설명 |
|---|---|
| book | 성경 권명 |
| chapter | 장 |
| verse | 절 |
| text | 성경 본문 |
| chunk_text | 검색에 사용되는 청크 본문 |
| reference | 구절 위치 |

---

### 3) 벡터 검색 기반 RAG 구조 구현

사용자 질문을 임베딩한 뒤, 성경 청크 데이터와 의미적으로 가까운 본문을 검색하는 구조를 구현했습니다.

전체 흐름은 다음과 같습니다.

```text
사용자 질문 입력
        ↓
질문 전처리 및 의도 파악
        ↓
질문 임베딩 생성
        ↓
벡터 DB에서 관련 성경 청크 검색
        ↓
검색 결과 검증
        ↓
근거 본문 기반 답변 생성
        ↓
사용자에게 답변 반환
```

---

### 4) 근거 기반 답변 생성

검색된 성경 본문을 LLM 프롬프트에 함께 전달하여 답변을 생성했습니다.

답변 생성 시 다음 원칙을 적용했습니다.

- 검색된 본문을 우선 근거로 사용
- 검색 결과가 부족하면 무리하게 답변하지 않음
- 성경 본문에 없는 내용은 단정하지 않음
- 교단별 해석 차이가 있을 수 있는 주제는 조심스럽게 표현
- 본문 근거가 약한 경우 `확인 불가` 형태로 응답

---

### 5) FastAPI 기반 챗봇 서버 구현

챗봇 기능을 API 형태로 사용할 수 있도록 FastAPI 서버를 구성했습니다.

주요 역할은 다음과 같습니다.

- 사용자 질문 수신
- RAG 검색 로직 실행
- LLM 답변 생성
- 결과 JSON 반환
- 웹 프론트엔드와 연동 가능한 구조 제공

예시 API 흐름은 다음과 같습니다.

```text
POST /chat
{
  "question": "사랑에 대해 성경은 뭐라고 말하나요?"
}
```

응답 예시는 다음과 같은 구조를 목표로 했습니다.

```json
{
  "answer": "성경에서는 사랑을 오래 참고 온유한 것으로 설명합니다...",
  "sources": [
    {
      "reference": "고린도전서 13장 4절",
      "text": "사랑은 오래 참고 사랑은 온유하며..."
    }
  ]
}
```

## 3. 프로젝트 진행 과정

### Step 1. 데이터 수집

Wikisource에서 성경 본문 데이터를 수집했습니다.  
초기에는 문서 구조가 일정하지 않아 HTML 파싱 방식과 텍스트 파싱 방식을 함께 검토했습니다.

수집 과정에서 다음과 같은 문제가 있었습니다.

- 페이지별 HTML 구조 차이
- 일부 본문에서 불필요한 문자 포함
- 장/절 구분 규칙 불일치
- API 응답 형태 차이

이를 해결하기 위해 본문 추출 규칙을 조정하고,  
수집 후 별도의 정제 과정을 거쳐 분석 가능한 데이터 형태로 변환했습니다.

---

### Step 2. 데이터 전처리

수집된 본문 데이터를 장, 절, 본문 단위로 정리했습니다.

전처리 과정에서는 다음 항목을 확인했습니다.

- 전체 행 수 확인
- 중복 구절 확인
- 빈 본문 확인
- 권명/장/절 누락 여부 확인
- 검색용 텍스트 정제

최종적으로 성경 구절 데이터를 CSV 형태로 저장했습니다.

---

### Step 3. 청크 생성

RAG 검색에 활용하기 위해 구절 단위 데이터를 청크 단위로 변환했습니다.

단일 구절만 사용할 경우 질문의 의미와 연결되는 문맥이 부족할 수 있기 때문에,  
주변 구절을 포함하는 방식으로 검색 단위를 구성했습니다.

이를 통해 질문과 관련된 성경 본문을 더 안정적으로 찾을 수 있도록 했습니다.

---

### Step 4. 벡터 DB 구축

생성된 청크 데이터를 임베딩하여 벡터 DB에 저장했습니다.

이후 사용자의 질문이 들어오면 질문을 임베딩하고,  
벡터 DB에서 의미적으로 유사한 성경 본문을 검색하도록 구성했습니다.

---

### Step 5. RAG 답변 로직 구현

검색된 본문을 LLM에 전달하여 답변을 생성했습니다.

이때 단순 생성형 답변이 아니라,  
검색된 구절을 기반으로 설명하도록 프롬프트를 구성했습니다.

또한 검색 결과가 부족한 경우에는 답변을 제한하도록 하여  
LLM의 환각 가능성을 줄이고자 했습니다.

---

### Step 6. FastAPI 서버 구현

RAG 로직을 API 서버로 실행할 수 있도록 FastAPI를 사용했습니다.

로컬 환경에서는 Uvicorn을 통해 서버를 실행하고,  
브라우저 또는 프론트엔드에서 질문을 보내 답변을 확인할 수 있도록 구성했습니다.

---

### Step 7. 배포 시도 및 한계 확인

Render를 활용한 배포를 시도했으나,  
무료 플랜의 메모리 제한으로 인해 서버 실행 중 메모리 초과 문제가 발생했습니다.

프로젝트 전체 파일 용량은 크지 않았지만,  
실제 서버 실행 시에는 다음 요소들이 메모리를 사용했습니다.

- Python 런타임
- FastAPI 서버
- 임베딩/벡터 검색 관련 라이브러리
- 벡터 DB 로딩
- LLM API 호출을 위한 패키지
- 데이터 파일 로딩

따라서 단순 파일 크기보다 **실행 중 메모리 사용량**이 배포 환경에서 중요한 요소임을 확인했습니다.

현재는 로컬 실행 및 GitHub 포트폴리오 정리를 중심으로 프로젝트를 정리했습니다.

## 6. 실행 방법

### 1) 가상환경 생성

```bash
python -m venv venv
```

Windows 기준 실행:

```bash
venv\Scripts\activate
```

---

### 2) 패키지 설치

```bash
pip install -r requirements.txt
```

---

### 3) 환경변수 설정

`.env` 파일을 생성하고 OpenAI API Key를 설정합니다.

```text
OPENAI_API_KEY=your_api_key_here
```

보안을 위해 실제 `.env` 파일은 GitHub에 업로드하지 않습니다.  
대신 `.env.example` 파일만 업로드합니다.

예시:

```text
OPENAI_API_KEY=
```

---

### 4) 서버 실행

```bash
uvicorn main:app --reload
```

또는 파일명이 `app.py`인 경우:

```bash
uvicorn app:app --reload
```

실행 후 브라우저에서 아래 주소로 접속합니다.

```text
http://127.0.0.1:8000
```

## 7. 답변 생성 방식

이 프로젝트의 답변 생성 방식은 다음과 같습니다.

```text
1. 사용자가 질문 입력
2. 질문을 임베딩 벡터로 변환
3. 벡터 DB에서 관련 성경 청크 검색
4. 검색된 구절과 질문을 함께 프롬프트에 입력
5. LLM이 검색된 본문을 근거로 답변 생성
6. 답변과 참고 구절을 함께 반환
```

이 구조를 통해 LLM이 단독으로 답변하는 방식보다  
본문 근거가 명확한 답변을 생성할 수 있도록 했습니다.

## 9. 프로젝트에서 고민한 부분

### 1) LLM 환각 방지

성경 챗봇은 잘못된 구절을 말하거나,  
성경에 없는 내용을 성경적 내용처럼 설명하면 신뢰도가 크게 떨어집니다.

따라서 검색 결과를 기반으로 답변하도록 설계하고,  
검색 결과가 부족한 경우 무리하게 답변하지 않도록 했습니다.

---

### 2) 검색 단위 설계

구절 단위 검색은 정확한 구절을 찾는 데 유리하지만,  
문맥이 부족할 수 있습니다.

반대로 너무 긴 청크는 검색 정확도가 떨어질 수 있습니다.

이 프로젝트에서는 구절과 주변 문맥을 함께 고려하는 청크 구조를 사용하여  
검색 정확도와 문맥 보존 사이의 균형을 맞추고자 했습니다.

---

### 3) 해석이 갈릴 수 있는 질문 처리

성경 질문 중에는 교단이나 신학적 관점에 따라 해석이 달라질 수 있는 주제가 있습니다.

따라서 단정적인 답변보다는,  
성경 본문에서 확인되는 범위 안에서 조심스럽게 답변하도록 설계했습니다.

---

### 4) 배포 환경의 메모리 제한

Render 무료 플랜에서 배포를 시도했으나 메모리 제한 문제가 발생했습니다.

이를 통해 RAG 프로젝트는 단순 코드 실행뿐 아니라,  
데이터 로딩, 벡터 DB, 서버 메모리, 라이브러리 사용량까지 고려해야 한다는 점을 확인했습니다.

## 11. 한계점

현재 프로젝트에는 다음과 같은 한계가 있습니다.

- 무료 배포 환경에서는 메모리 제한으로 안정적인 서비스 운영이 어려움
- 성경 본문 검색 결과의 품질이 질문 표현 방식에 따라 달라질 수 있음
- 신학적 해석이 필요한 질문은 단순 본문 검색만으로 충분하지 않을 수 있음
- 사용자 질문 로그 기반의 개선 루프는 아직 적용하지 않음
- 다양한 번역본 비교 기능은 포함하지 않음

## 12. 향후 개선 방향

향후 다음과 같은 기능을 추가할 수 있습니다.

### 1) 질문 유형 분류 고도화

질문을 다음과 같이 분류하여 검색 전략을 다르게 적용할 수 있습니다.

- 구절 검색형 질문
- 주제 설명형 질문
- 인물/사건 질문
- 교리/신학 질문
- 적용/상담형 질문

---

### 2) 해설 레이어 추가

성경 본문만으로 답변하기 어려운 질문을 위해  
별도의 해설 데이터 또는 QA 가이드 데이터를 추가할 수 있습니다.

예시:

```text
qa_guides.csv
- topic
- question_pattern
- answer_guide
- related_verses
- caution
```

이를 통해 단순 구절 검색을 넘어  
더 안정적인 해설형 답변을 제공할 수 있습니다.

---

### 3) 검색 평가 데이터셋 구축

대표 질문과 기대 구절을 정리한 평가셋을 만들어  
검색 성능을 정량적으로 확인할 수 있습니다.

예시 평가 항목:

- Top-1 검색 정확도
- Top-3 검색 정확도
- 질문별 관련 구절 포함 여부
- 답변의 근거 일치 여부

---

### 4) 배포 환경 개선

안정적인 서비스 운영을 위해 다음 방식을 검토할 수 있습니다.

- 유료 서버 사용
- Docker 기반 배포
- 벡터 DB 경량화
- 데이터 로딩 방식 최적화
- 서버 시작 시 필요한 파일만 로드하도록 개선

---

### 5) 사용자 피드백 기반 개선

사용자가 답변에 대해 피드백을 남길 수 있도록 하고,  
이를 기반으로 검색 품질과 답변 품질을 개선할 수 있습니다.

예시:

```text
좋아요 / 별로예요
관련 구절이 맞는지 여부
답변이 충분했는지 여부
추가로 궁금한 질문
```

## 13. 프로젝트를 통해 배운 점

이 프로젝트를 통해 단순히 LLM API를 호출하는 것과  
실제 서비스 형태의 RAG 챗봇을 구현하는 것은 다르다는 점을 경험했습니다.

특히 다음과 같은 부분을 직접 다뤘습니다.

- 원천 데이터 수집과 정제의 중요성
- 검색 단위인 청크 설계의 중요성
- LLM 답변보다 검색 근거의 품질이 중요하다는 점
- 환각 방지를 위한 응답 제한 규칙의 필요성
- 로컬 실행과 실제 배포 환경의 차이
- RAG 시스템에서도 데이터, 검색, 서버, 비용을 함께 고려해야 한다는 점

## 14. 포트폴리오 관점의 의미

이 프로젝트는 단순 챗봇 제작이 아니라,  
도메인 데이터 기반 RAG 시스템을 직접 설계하고 구현한 프로젝트입니다.

데이터 수집부터 전처리, 청크 생성, 벡터 검색, LLM 답변 생성, API 서버 구성까지  
RAG 서비스의 전체 흐름을 경험했다는 점에서 의미가 있습니다.

특히 다음 역량을 보여줄 수 있습니다.

- 비정형 텍스트 데이터 수집 및 정제
- RAG 파이프라인 설계
- 벡터 검색 기반 질의응답 구현
- LLM 환각 방지 로직 설계
- FastAPI 기반 API 서버 구현
- 배포 환경의 제약 분석 및 개선 방향 도출

## 17. 마무리

Bible RAG Chatbot은 성경 본문이라는 도메인 데이터를 기반으로  
근거 중심의 질의응답 시스템을 구현한 프로젝트입니다.

LLM의 생성 능력에만 의존하지 않고,  
검색된 성경 본문을 기반으로 답변하도록 구성하여  
더 신뢰도 높은 챗봇 구조를 만들고자 했습니다.

현재는 로컬 실행과 포트폴리오 정리를 중심으로 완성했으며,  
향후 검색 평가, 해설 데이터 추가, 배포 최적화를 통해 더 발전시킬 수 있습니다.
